**AUDIT-5: Goroutine/channel leak during reconnect**

The problem: When `reconnect()` replaces `s.writeCh` with a new channel, old `taskLoop` goroutines still hold references to the old `writeCh`. They may block forever trying to send on a channel nobody reads.

Fix approach: In `recovery.go`, the reconnect already waits for `<-oldDone` which ensures the `dispatchLoop` exits. But individual `taskLoop` goroutines may still be running. The fix is to cancel all snapshotted tasks' contexts before replacing channels. The current code already snapshots tasks and unregisters ITTs. Add explicit task cancellation to ensure old taskLoops exit:

1. In `reconnect()`, after snapshotting tasks at Step 2, add:
```go
// Cancel all old taskLoop goroutines by closing their PDU channels.
// Unregistering from router closes the PDU channel, which causes
// taskLoop to exit its range loop.
for itt := range taskSnapshot {
    s.router.Unregister(itt)
}
```

This is already done in the current code. The issue is that taskLoop goroutines may be blocked on `writeCh <- raw` (sending Data-Out or SNACK). To fix this:

2. Access writeCh via a method that returns the current channel under lock. Change the `sendSNACK` and Data-Out sending code to use `s.getWriteCh()` instead of capturing `writeCh` directly:

Add to `session.go`:
```go
// getWriteCh returns the current write channel under lock.
// This ensures goroutines always send on the current (not stale) channel.
func (s *Session) getWriteCh() chan<- *transport.RawPDU {
    s.mu.Lock()
    defer s.mu.Unlock()
    return s.writeCh
}
```

3. Update `sendSNACK` in `snack.go` to accept a `func() chan<- *transport.RawPDU` getter instead of a `chan<- *transport.RawPDU`:

Change the signature:
```go
func (t *task) sendSNACK(getWriteCh func() chan<- *transport.RawPDU, snackType uint8, begRun, runLength uint32, expStatSN uint32) error {
```

And the send:
```go
    writeCh := getWriteCh()
    select {
    case writeCh <- raw:
        return nil
    default:
        return fmt.Errorf("session: writeCh full, cannot send SNACK")
    }
```

4. Update all callers of `sendSNACK` to pass `s.getWriteCh` instead of `s.writeCh`.

5. Similarly update `startSnackTimer` and `resetSnackTimer` to accept the getter.

**AUDIT-6: Blocking SNACK send with context timeout**

In `sendSNACK`, replace the non-blocking send with a context-aware blocking send:

```go
func (t *task) sendSNACK(getWriteCh func() chan<- *transport.RawPDU, snackType uint8, begRun, runLength uint32, expStatSN uint32) error {
    snack := &pdu.SNACKReq{
        Header: pdu.Header{
            Final:            true,
            InitiatorTaskTag: t.itt,
        },
        Type:      snackType,
        BegRun:    begRun,
        RunLength: runLength,
        ExpStatSN: expStatSN,
    }

    bhs, err := snack.MarshalBHS()
    if err != nil {
        return fmt.Errorf("session: encode SNACK: %w", err)
    }

    raw := &transport.RawPDU{BHS: bhs}
    
    // Use blocking send with 5-second timeout per RFC 7143 Section 11.16
    // which requires reliable SNACK transmission.
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    
    writeCh := getWriteCh()
    select {
    case writeCh <- raw:
        return nil
    case <-ctx.Done():
        return fmt.Errorf("session: SNACK send timed out after 5s (writeCh blocked)")
    }
}
```

Add `"context"` and `"time"` to snack.go imports.

Log at Error level when SNACK send fails:
```go
if err := t.sendSNACK(...); err != nil {
    s.cfg.logger.Error("session: SNACK delivery failed", "itt", t.itt, "error", err)
}
```

**AUDIT-7: Keep old ITT registered until reassignment confirmation**

The problem: In `replaceConnection()` at lines 113-142, the old ITT is unregistered from the router BEFORE the TMF TASK REASSIGN completes. This means any target responses to the old ITT during the reassignment window are lost.

Fix: Restructure the task reassignment loop to keep old ITT registered until after TMF confirmation:

```go
for itt, tk := range taskSnapshot {
    // Allocate new ITT for reassigned task BEFORE unregistering old one.
    newITT := s.router.AllocateITT()
    newPduCh := s.router.RegisterPersistent(newITT)

    // Send TASK REASSIGN TMF referencing the OLD ITT.
    // The old ITT remains registered so any in-flight target responses
    // are still routed correctly during the reassignment window.
    tmfResult, tmfErr := s.sendTMF(newCtx, TMFTaskReassign, itt, tk.lun)
    if tmfErr != nil || (tmfResult != nil && tmfResult.Response != TMFRespComplete) {
        // Reassign failed -- fail this task.
        respStr := "send error"
        if tmfResult != nil {
            respStr = fmt.Sprintf("response=%d", tmfResult.Response)
        }
        // Clean up the new ITT we allocated but won't use.
        s.router.Unregister(newITT)
        // NOW unregister the old ITT since we're giving up.
        s.router.Unregister(itt)
        tk.cancel(fmt.Errorf("session: task reassign failed: %s: %v", respStr, tmfErr))
        continue
    }

    // TMF TASK REASSIGN confirmed -- NOW safe to unregister old ITT.
    s.router.Unregister(itt)

    // Update task with new ITT.
    s.mu.Lock()
    tk.itt = newITT
    s.tasks[newITT] = tk
    delete(s.tasks, itt)
    s.mu.Unlock()

    // Restart task loop on new channel.
    go s.taskLoop(tk, newPduCh)
}
```

Key changes from original:
1. `s.router.Unregister(itt)` moved AFTER `sendTMF` succeeds (was BEFORE).
2. On failure, both old and new ITTs are cleaned up.
3. Variable renamed from `pduCh` to `newPduCh` for clarity.

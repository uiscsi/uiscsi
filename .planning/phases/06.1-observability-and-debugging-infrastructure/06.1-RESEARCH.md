# Phase 06.1: Observability and Debugging Infrastructure - Research

**Researched:** 2026-04-01
**Domain:** Go stdlib observability (log/slog, hash/crc32, fmt.Stringer), callback hooks, structured errors
**Confidence:** HIGH

## Summary

This phase adds observability instrumentation to an existing, well-structured Go codebase with zero external dependencies. All required functionality (CRC32C verification, slog logging, fmt.Stringer, typed errors, callback hooks) uses only Go stdlib. The codebase already has the building blocks: `HeaderDigest()`/`DataDigest()` computation functions, `WithLogger(*slog.Logger)` session option, `WithAsyncHandler` callback pattern, and `DecodeBHS()` for typed PDU dispatch. The work is connecting these pieces and adding String() methods.

The primary risk is performance regression in hot paths (ReadPump/WritePump). PDU hooks and slog calls must not allocate on the hot path when disabled. slog's `Logger.Enabled()` check and nil-function-pointer guards are the standard mitigation.

**Primary recommendation:** Follow the existing functional options pattern (`SessionOption`) for all new hooks. Keep digest verification in `ReadRawPDU` (transport layer), keep hooks and slog in pump functions (transport layer with injected logger), and keep `String()` methods on typed PDU structs (pdu package). No new packages needed.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Strict RFC 7143 Section 7.3 compliance -- fail connection immediately on CRC32C mismatch
- **D-02:** `ReadRawPDU` verifies digests inline when digest flags are set; returns typed `DigestError` with expected value, actual value, and which digest (header/data)
- **D-03:** ReadPump treats `DigestError` as connection-fatal (ERL 0 behavior). SNACK-based recovery for ERL 1+ can use the error type to distinguish digest failures from other read errors
- **D-04:** Callback hooks via `WithPDUHook(func(direction, *RawPDU))` session option, matching existing `WithAsyncHandler` pattern
- **D-05:** Hook is called in ReadPump (after read, before dispatch) and WritePump (before write) so tests can record full PDU sequences
- **D-06:** Hook sees `*RawPDU` (raw wire bytes) -- callers decode if they need typed fields
- **D-07:** All 18 PDU types get `String()` methods returning key fields in compact format: `SCSICommand{OpCode:0x01 ITT:0x0001 CmdSN:42 LUN:0 CDB:28:00:...}`
- **D-08:** Format is struct-name + curly braces with space-separated key:value pairs. Hex for opcodes/ITTs, decimal for sequence numbers, truncated hex for CDB/data
- **D-09:** Push-based callback via `WithMetricsHook(func(MetricEvent))` session option
- **D-10:** `MetricEvent` is a typed enum covering: PDUSent, PDUReceived (with opcode), CommandComplete (with latency), BytesIn, BytesOut
- **D-11:** No concrete stats struct -- consumers aggregate as they see fit. Library stays dependency-free
- **D-12:** Debug = every PDU send/receive with opcode, ITT, CmdSN, StatSN, data lengths
- **D-13:** Info = login stage transitions, session open/close, reconnection start/complete, command window changes
- **D-14:** Warn = digest mismatches (before connection drop), dropped PDUs, timeout retries
- **D-15:** Error = unrecoverable failures only
- **D-16:** All slog calls use the session's injected `*slog.Logger` (via `WithLogger`), never the global `slog` package directly -- except in transport/pump.go which has no session context (convert to accept logger parameter)

### Pattern 1: DigestError as Typed Error
**What:** A concrete error type in `internal/digest` that carries expected vs actual digest values and which digest failed (header/data).
**When to use:** Returned by `ReadRawPDU` when digest verification fails. Checked by ReadPump via `errors.As` to distinguish digest errors from I/O errors.
**Example:**
```go
// internal/digest/errors.go
package digest

import "fmt"

// DigestType indicates whether a header or data digest failed.
type DigestType uint8

const (
    DigestHeader DigestType = iota
    DigestData
)

func (d DigestType) String() string {
    switch d {
    case DigestHeader:
        return "header"
    case DigestData:
        return "data"
    default:
        return "unknown"
    }
}

// DigestError indicates a CRC32C digest mismatch on a received PDU.
// Per RFC 7143 Section 7.3, this is a connection error at ERL 0.
type DigestError struct {
    Type     DigestType
    Expected uint32
    Actual   uint32
}

func (e *DigestError) Error() string {
    return fmt.Sprintf("digest: %s CRC32C mismatch: expected 0x%08X, got 0x%08X",
        e.Type, e.Expected, e.Actual)
}
```

### Pattern 2: PDU Hook Callback
**What:** A function passed via `WithPDUHook` that is called on every PDU send/receive with direction and raw PDU bytes.
**When to use:** Tests record full PDU sequences for protocol compliance verification.
**Example:**
```go
// Direction type for PDU hooks
type PDUDirection uint8

const (
    PDUSend    PDUDirection = iota
    PDUReceive
)

// In sessionConfig:
// pduHook func(PDUDirection, *transport.RawPDU)

// WithPDUHook registers a callback invoked for every PDU sent or received.
func WithPDUHook(h func(PDUDirection, *transport.RawPDU)) SessionOption {
    return func(c *sessionConfig) {
        c.pduHook = h
    }
}
```

### Pattern 3: MetricEvent as Struct with Type Field
**What:** A single struct with a type discriminator field, rather than an interface or sum type. This is the simplest Go pattern for discriminated unions without generics overhead.
**When to use:** For the push-based metrics callback (D-09/D-10/D-11).
**Example:**
```go
// MetricEventType discriminates metric event kinds.
type MetricEventType uint8

const (
    MetricPDUSent         MetricEventType = iota
    MetricPDUReceived
    MetricCommandComplete
    MetricBytesIn
    MetricBytesOut
)

// MetricEvent carries a single metric observation.
type MetricEvent struct {
    Type    MetricEventType
    OpCode  pdu.OpCode // for PDUSent/PDUReceived
    Bytes   uint64     // for BytesIn/BytesOut
    Latency time.Duration // for CommandComplete
}
```
**Rationale:** A struct-with-type-field avoids interface boxing allocations, keeps the API simple, and allows consumers to switch on Type. Fields irrelevant to a given Type are zero-valued, which is harmless. This is simpler than an interface hierarchy and more Go-idiomatic than a sealed sum type.

### Pattern 4: Logger Injection into Transport Pumps
**What:** `ReadPump` and `WritePump` currently use `slog.Warn` (global logger). Per D-16, they must accept a `*slog.Logger` parameter.
**When to use:** The session layer passes its configured logger when calling pump functions.
**Example:**
```go
// Updated ReadPump signature:
func ReadPump(ctx context.Context, r io.Reader, router *Router,
    unsolicitedCh chan<- *RawPDU, digestHeader, digestData bool,
    logger *slog.Logger, pduHook func(PDUDirection, *RawPDU)) error {
    // ...
    if logger.Enabled(ctx, slog.LevelDebug) {
        logger.DebugContext(ctx, "transport: PDU received",
            "opcode", opcode.String(),
            "itt", itt,
            "dsLen", dsLen)
    }
}
```
**Note:** The `Logger.Enabled()` check is critical for hot-path performance. When Debug is not enabled, the only cost is a single boolean comparison -- no attribute allocation occurs.

### Pattern 5: String() on Typed PDU Structs
**What:** Each of the 18 typed PDU structs implements `fmt.Stringer` with the format from D-07/D-08.
**When to use:** For slog Debug logging, test output, protocol debugging.
**Example:**
```go
func (p *SCSICommand) String() string {
    return fmt.Sprintf("SCSICommand{OpCode:0x%02x ITT:0x%04x CmdSN:%d LUN:%d CDB:%s}",
        uint8(OpSCSICommand),
        p.InitiatorTaskTag,
        p.CmdSN,
        binary.BigEndian.Uint64(p.LUN[:]),
        truncHex(p.CDB[:], 8))
}

// truncHex returns colon-separated hex bytes, truncated to maxBytes.
func truncHex(b []byte, maxBytes int) string {
    if len(b) > maxBytes {
        b = b[:maxBytes]
    }
    // format as "28:00:00:00:..."
    // ...
}
```

### Anti-Patterns to Avoid
- **Allocating slog attributes when logging is disabled:** Always guard with `logger.Enabled(ctx, level)` before constructing attribute slices in hot paths (ReadPump/WritePump).
- **Holding locks while calling hooks:** PDU hooks and metric hooks must be called outside any session mutex to prevent deadlocks if the callback does synchronous work.
- **Modifying RawPDU in hooks:** The hook receives a pointer to the same RawPDU being dispatched. Document that hooks MUST NOT modify the PDU. Alternatively, could pass a copy but that adds allocation cost -- pointer with documentation is the Go convention.
- **Using `slog.Default()` in transport layer:** Per D-16, transport functions must accept an explicit `*slog.Logger` parameter. Fallback to `slog.Default()` only in `defaultConfig()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CRC32C computation | Custom CRC implementation | `hash/crc32` with `crc32.Castagnoli` | Already implemented in `internal/digest/crc32c.go`. Hardware-accelerated on amd64. |
| Structured logging | Custom logger | `log/slog` | Already wired via `WithLogger`. Zero-cost disabled-level check via `Logger.Enabled()`. |
| Hex formatting | Custom hex encoder | `encoding/hex` or `fmt.Sprintf("%x")` | Stdlib handles this cleanly. |

**Key insight:** This phase is pure wiring -- connecting existing digest computation to verification, extending existing slog injection to more call sites, and adding `String()` methods. No new algorithms or complex data structures needed.

## Common Pitfalls

### Pitfall 1: Hot-Path Allocation in Disabled Logging
**What goes wrong:** slog attribute construction allocates even when the level is disabled, causing GC pressure on every PDU.
**Why it happens:** `logger.Debug("msg", "key", value)` evaluates arguments before checking level.
**How to avoid:** Use `logger.Enabled(ctx, slog.LevelDebug)` guard before any slog call in ReadPump/WritePump. For slog.LogValuer types, slog defers evaluation, but explicit guards are clearer.
**Warning signs:** Unexplained allocation increase in benchmark tests.

### Pitfall 2: Breaking ReadRawPDU Signature
**What goes wrong:** Adding digest verification to `ReadRawPDU` changes its return semantics -- existing callers (login phase) may not expect `DigestError`.
**Why it happens:** Login phase uses `ReadRawPDU` directly before digests are negotiated, so `digestHeader=false, digestData=false` means no verification occurs. But callers should still be checked.
**How to avoid:** Digest verification only runs when `digestHeader` or `digestData` is true. Existing callers pass `false` during login, so no behavioral change. Verify all call sites.
**Warning signs:** Login tests failing after framer changes.

### Pitfall 3: ReadPump/WritePump Signature Change Breaks Callers
**What goes wrong:** Adding `logger` and `pduHook` parameters to `ReadPump`/`WritePump` breaks all existing call sites (session, tests).
**Why it happens:** These are package-level functions with many callers.
**How to avoid:** Update all call sites in a single commit. Alternatively, use an options struct, but given Go convention and that these are internal functions, extra parameters are acceptable.
**Warning signs:** Compilation failures in session and test packages.

### Pitfall 4: Deadlock from Hook Calling Back into Session
**What goes wrong:** If a PDU hook or metrics hook calls a session method that acquires the session mutex, and the hook is called while the mutex is held, deadlock.
**Why it happens:** ReadPump/WritePump run as goroutines but hooks bridge back to user code.
**How to avoid:** Hooks are called from pump goroutines which do NOT hold the session mutex. The session only holds its mutex briefly for task map operations. As long as hooks are called outside any lock, this is safe. Document that hooks must not call session methods.
**Warning signs:** Test hangs.

### Pitfall 5: MetricEvent Latency Measurement Requires Start Time
**What goes wrong:** CommandComplete latency needs a start time, but the metrics hook only fires at completion.
**Why it happens:** `Submit()` creates the task but the metrics layer doesn't know when the command started.
**How to avoid:** Record `time.Now()` in the task struct at submission time. When the task completes (in `taskLoop`), compute `time.Since(startTime)` and fire the MetricCommandComplete event.
**Warning signs:** Zero or garbage latency values.

### Pitfall 6: Digest Verification Must Include Padding
**What goes wrong:** Data digest is computed over data + zero padding, but verification only checks raw data bytes.
**Why it happens:** RFC 7143 Section 12.1 specifies digest covers data segment INCLUDING padding bytes.
**How to avoid:** `DataDigest()` in `internal/digest/crc32c.go` already includes padding in computation. Use the same function for verification -- compute `DataDigest(raw.DataSegment)` and compare with `raw.DataDigest`.
**Warning signs:** Data digest failures on non-4-byte-aligned payloads.

## Code Examples

### Digest Verification in ReadRawPDU
```go
// After reading and parsing the PDU, before returning:
if digestHeader {
    // Compute expected header digest over BHS + AHS
    input := make([]byte, pdu.BHSLength+len(raw.AHS))
    copy(input, raw.BHS[:])
    copy(input[pdu.BHSLength:], raw.AHS)
    expected := digest.HeaderDigest(input)
    if expected != raw.HeaderDigest {
        PutBuffer(payload)
        return nil, &digest.DigestError{
            Type:     digest.DigestHeader,
            Expected: expected,
            Actual:   raw.HeaderDigest,
        }
    }
}
if digestData && dsLen > 0 {
    expected := digest.DataDigest(raw.DataSegment)
    if expected != raw.DataDigest {
        PutBuffer(payload)
        return nil, &digest.DigestError{
            Type:     digest.DigestData,
            Expected: expected,
            Actual:   raw.DataDigest,
        }
    }
}
```

### slog Debug Logging in ReadPump
```go
// After successful ReadRawPDU, before dispatch:
opcode := pdu.OpCode(raw.BHS[0] & 0x3f)
```

## Design Recommendations (Claude's Discretion Areas)

### MetricEvent: Struct with Type Field (Recommended)
Use a struct with a `MetricEventType` discriminator. Rationale:
- No interface boxing allocations
- Single allocation per event (the struct itself, stack-allocated if hook is inlined)
- Clean `switch` on Type in consumer code
- Simpler than interface hierarchy; more Go-idiomatic

### slog Attribute Naming: snake_case (Recommended)
Use snake_case for slog attribute keys: `cmd_sn`, `stat_sn`, `itt`, `ds_len`, `opcode`. Rationale:
- slog stdlib examples use snake_case
- Consistent with Go's lowercase convention
- More readable than camelCase in log output
- Matches the existing `"opcode"` key already used in pump.go

### String() on Typed PDU Structs (Confirmed)
Place `String()` methods on the typed structs in `initiator.go` and `target.go` (or a new `stringer.go` in the pdu package). The CONTEXT.md notes "typed structs preferred" and this is correct because:
- slog will call `String()` automatically via `fmt.Stringer`
- DecodeBHS already returns typed structs
- RawPDU is a transport-layer concern; String() is a presentation concern

### Internal Helper Organization
- `internal/digest/errors.go` -- new file for DigestError (small, focused)
- `internal/pdu/stringer.go` -- new file for all 18 String() methods + truncHex helper (keeps initiator.go/target.go focused on marshal/unmarshal)
- `internal/session/metrics.go` -- new file for MetricEvent type and helper functions
- Transport pump.go gets modified in place (parameter additions + hook/logging calls)
- Session types.go gets new option functions; session.go gets hook/metrics plumbing

## Open Questions

1. **Header digest input scope**
   - What we know: RFC 7143 Section 12.1 says header digest covers BHS + AHS. The existing `HeaderDigest()` function takes a byte slice.
   - What's unclear: When AHS is nil (most PDUs), should we pass just BHS or BHS+empty? Answer: just BHS bytes -- the function operates on whatever bytes it receives.
   - Recommendation: `HeaderDigest(raw.BHS[:])` when no AHS, `HeaderDigest(append(raw.BHS[:], raw.AHS...))` when AHS present.

2. **Metrics hook threading model**
   - What we know: Hooks are called from pump goroutines (ReadPump/WritePump) and from taskLoop goroutines.
   - What's unclear: Should the metrics hook be thread-safe or documented as called from multiple goroutines?
   - Recommendation: Document that the hook MAY be called concurrently from multiple goroutines. Consumer is responsible for synchronization. This matches Go convention (e.g., `http.Handler`).

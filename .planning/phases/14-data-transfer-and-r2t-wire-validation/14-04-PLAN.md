Create test/conformance/r2t_test.go with package `conformance_test`.

All tests use: ImmediateData=No, InitialR2T=Yes on both sides (pure R2T-driven writes). This isolates R2T fulfillment from unsolicited data paths.

**Test 1: TestR2T_SinglePDU** (R2T-01, FFP #12.1)
- HandleSCSIFunc: send 1 R2T with TTT=0x100, BufferOffset=0, DesiredDataTransferLength=512. ReadDataOutPDUs (expect 1 PDU). Send SCSIResponse.
- Write 512 bytes.
- Assert from captured Data-Out: exactly 1 PDU. TTT=0x100. BufferOffset=0. DataSN=0. Final=true. DataSegmentLen=512.

**Test 2: TestR2T_MultiPDU** (R2T-02, FFP #12.2)
- Config: MaxRecvDSL=256 on target.
- HandleSCSIFunc: send 1 R2T with TTT=0x200, BufferOffset=0, DesiredDTL=1024. ReadDataOutPDUs (expect 4 PDUs). Send SCSIResponse.
- Write 1024 bytes.
- Assert from captured Data-Out: 4 PDUs, all with TTT=0x200. DataSN: 0,1,2,3. BufferOffset: 0,256,512,768. Final: false,false,false,true. Each DataSegmentLen=256.

**Test 3: TestR2T_MultipleR2T** (R2T-03, FFP #12.3)

**IMPORTANT — DataSequenceInOrder=No gap:** FFP #12.3 specifies testing with DataSequenceInOrder=No. However, the initiator hardcodes DataSequenceInOrder="Yes" in its login proposals (internal/login/login.go:529), and this key uses BooleanOr negotiation semantics (internal/login/negotiation.go:54), meaning the result is "Yes" if either side proposes "Yes". It is therefore impossible to negotiate DataSequenceInOrder=No with the current initiator. True out-of-order R2T fulfillment testing requires a future initiator change to support proposing DataSequenceInOrder=No.

**What this test DOES verify (partial R2T-03 coverage under DataSequenceInOrder=Yes):**
- Target sends 2 R2Ts for the same command with non-contiguous buffer offsets (simulating what a target might send):
  - R2T 0: TTT=0x300, BufferOffset=0, DesiredDTL=512, R2TSN=0
  - R2T 1: TTT=0x301, BufferOffset=512, DesiredDTL=512, R2TSN=1
  ReadDataOutPDUs for first burst (F-bit). ReadDataOutPDUs for second burst (F-bit). Send SCSIResponse.
- Write 1024 bytes.
- Assert: captured Data-Out PDUs group by TTT. Each burst has DataSN starting at 0 (per-burst reset). First burst TTT=0x300 with BufferOffset starting at 0. Second burst TTT=0x301 with BufferOffset starting at 512.
- Assert: bursts are fulfilled in sequential order (R2TSN=0 before R2TSN=1) since DataSequenceInOrder=Yes.
- Add a `t.Log` note: "R2T-03 partial: DataSequenceInOrder=No cannot be negotiated (initiator hardcodes Yes, BooleanOr semantics). This test verifies multi-R2T per-burst isolation under DataSequenceInOrder=Yes. Full R2T-03 coverage requires initiator support for DataSequenceInOrder=No proposal."

**Test 4: TestR2T_ParallelCommand** (R2T-04, FFP #12.4)
- HandleSCSIFunc uses callCount to handle 2 commands:
  - callCount==0: send R2T with TTT=0x400, BufferOffset=0, DesiredDTL=512 for first command. ReadDataOutPDUs. Send SCSIResponse.
  - callCount==1: send R2T with TTT=0x500, BufferOffset=0, DesiredDTL=512 for second command. ReadDataOutPDUs. Send SCSIResponse.
- Launch 2 WriteBlocks concurrently (goroutines or sequential, 512 bytes each).
- Assert: captured Data-Out PDUs with TTT=0x400 have ITT matching first command. Data-Out PDUs with TTT=0x500 have ITT matching second command. Each set has DataSN=0, Final=true.
- For ITT correlation: captured SCSI Command PDUs (rec.Sent(pdu.OpSCSICommand)) carry the ITT. Match command ITT to Data-Out ITT.
